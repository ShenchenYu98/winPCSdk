## 一、架构概览

### 1.1 系统定位

Skill SDK 是运行在 IM 客户端内的一层客户端中枢，负责把 IM 聊天窗口、Skill 小程序与 Skill Server 连接起来，并对外提供统一的会话、流式事件、状态回调、历史恢复与结果回传能力。

本期架构聚焦 **SDK 本体**，外部系统一律作为依赖，不展开其内部实现。

### 1.2 设计原则

1. **SDK 为核心**：架构围绕 SDK 自身能力边界组织。
2. **统一接口**：IM 客户端与 Skill 小程序复用同一套 SDK 能力。
3. **连接复用**：WebSocket 长连接按用户级复用，不按会话重复建连。
4. **事件对齐**：`StreamMessage` 尽量与 Skill Server WebSocket 协议保持一致。
5. **状态可解释**：`onSessionStatusChange` 必须能追溯到原始流式事件。
6. **恢复优先连续性**：恢复页面时既要能拿到历史消息，也要能接住执行中的增量流。
7. **低侵入集成**：SDK 不替代 IM UI 和小程序 UI，只提供支撑能力。

### 1.3 外部依赖

| 依赖系统 | 角色 | SDK 依赖内容 |
|------|------|------|
| IM 客户端 | 宿主环境 | 触发 Skill、展示 Mini Bar、承载 SDK |
| Skill 小程序 | UI 容器 | 使用 SDK 发送消息、接收流式事件、渲染对话 |
| Skill Server | 后端能力提供方 | REST API、WebSocket 推流、会话与消息持久化 |
| IM 平台 API | 外部能力 | 通过 Skill Server 完成消息发送到聊天 |

### 2.2 架构分层

| 层级 | 模块 | 职责 |
|------|------|------|
| 接入层 | SDK Facade | 对外暴露统一 API |
| 编排层 | Session Orchestrator | 串联建连、查会话、建会话、首条消息发送、停止、重发 |
| 连接层 | Stream Connection Manager | 管理 WebSocket 生命周期、连接复用、监听器分发 |
| 协议层 | Event Router / Normalizer | 接收服务端原始事件并转换为 SDK 可消费结构 |
| 状态层 | Session Status Mapper | 将流式事件映射为 `executing / completed / stopped` |
| 缓存层 | Message Cache Store | 维护流式缓存、恢复数据、历史消息合并结果 |
| UI 桥接层 | Miniapp Bridge | 为 IM 客户端和 Skill 小程序提供状态回调与控制能力 |

## 三、核心模块设计

### 3.1 SDK Facade

对外提供统一入口，屏蔽 REST / WebSocket 细节。  
核心接口包括：

- `createSession`、`closeSkill`、`stopSkill`
- `sendMessage`、`regenerateAnswer`
- `getSessionMessage`
- `registerSessionListener`、`unregisterSessionListener`
- `onSessionStatusChange`、`onSkillWecodeStatusChange`
- `replyPermission`、`sendMessageToIM`、`controlSkillWeCode`

### 3.2 Session Orchestrator

负责会话编排：

- 首次触发时按 `imGroupId + ak + ACTIVE` 查找可复用会话
- 无可复用会话时创建会话
- 查询结果存在多个会话时，按服务端返回原始顺序取第一个
- 拿到 `welinkSessionId` 后由上层继续调用 `sendMessage` 发送首条消息
- 统一处理停止、重新生成、多轮消息发送

**关键约束**：

- `createSession` 只负责创建或复用会话并建立连接，不触发首轮执行
- 首轮执行由 `sendMessage` 触发
- 小程序内后续多轮统一走 `sendMessage`
- 会话上下文以 `welinkSessionId` 为核心标识
- Mini Bar 与小程序使用的会话 ID 由 IM 客户端管理和传入

### 3.3 Stream Connection Manager

负责连接管理：

- WebSocket 按用户级复用
- 监听器支持先注册后建连
- 当注册时连接尚未建立，先缓存注册信息，待连接建立后自动生效
- 按 `welinkSessionId` 分发流式事件
- `closeSkill` 只释放本地连接资源，不清理消息缓存

### 3.4 Event Router / Event Normalizer

负责事件接入与统一转换：

- 接收 `text.delta`、`thinking.delta`、`tool.update`、`question`、`permission.ask`、`snapshot`、`streaming` 等事件
- 输出统一 `StreamMessage`
- 为状态映射、缓存合并、UI 渲染提供标准输入

### 3.5 Session Status Mapper

负责状态解释：

- `step.start`、`text.delta`、`thinking.delta`、`tool.update`、`question`、`permission.ask`、`file`、`session.status=busy/retry` -> `executing`
- `step.done`、`text.done`、`thinking.done`、`session.status=idle` -> `completed`
- `session.error`、`error`、`agent.offline`、`permission.reply=reject` -> `stopped`

### 3.6 Message Cache Store

负责本地缓存与恢复：

- 以 `welinkSessionId + messageId + partId` 为缓存键
- 合并未落库的增量消息
- 接住 `snapshot` / `streaming` 恢复事件
- 为 `getSessionMessage` 和 `sendMessageToIM` 提供共享缓存能力
- `closeSkill` 后缓存保留
- 小程序关闭后缓存保留
- 应用重启后缓存不可用
- 服务端历史消息是权威来源，本地缓存仅补齐未落库或进行中的内容

### 3.7 Miniapp Bridge

负责宿主与小程序协同：

- 提供关闭/最小化控制
- 提供状态变化回调
- 支持 IM 聊天窗口与 Skill 小程序之间的展开/收起切换

## 四、外部接口协作关系

### 4.1 与 Skill Server

| 通道 | 方向 | 用途 |
|------|------|------|
| REST | SDK -> Skill Server | 会话查询/创建、消息发送、权限回复、历史查询、发送到 IM |
| WebSocket | Skill Server -> SDK | 流式消息推送、状态事件、恢复事件 |

### 4.2 与 IM 客户端

| 场景 | IM 客户端职责 | SDK 职责 |
|------|------|------|
| "/" 触发 | 展示 Skill 入口与输入框 | 承接会话创建/复用 |
| 首轮生成 | 管理当前 `welinkSessionId` 并发起首条消息 | 建连、复用会话、发送消息 |
| Mini Bar | 提供状态条 UI | 提供状态回调和执行态 |
| 展开/收起 | 宿主窗口控制，并向小程序传入 `welinkSessionId` | 提供小程序生命周期控制能力 |

### 4.3 与 Skill 小程序

| 场景 | 小程序职责 | SDK 职责 |
|------|------|------|
| 多轮对话 | 渲染对话视图和交互组件 | 提供消息发送、流式监听、历史恢复 |
| 权限与问题 | 渲染确认 UI / 提问 UI | 提供事件和回复能力 |
| 选文发送到 IM | 获取最终确认文本对应的消息上下文 | 提供按会话/消息标识发送能力 |

## 五、核心时序与数据流

### 5.1 首次触发时序

1. IM 客户端调用 `createSession`
2. SDK 建立或复用 WebSocket
3. SDK 按 `imGroupId + ak + ACTIVE` 查询可复用会话
4. 若无可复用会话，调用服务端创建会话
5. 若有多个可复用会话，按服务端返回原始顺序取第一个
6. SDK 返回 `welinkSessionId`
7. IM 客户端注册 `registerSessionListener` / `onSessionStatusChange`
8. IM 客户端调用 `sendMessage` 发送首条 `content`
9. Skill Server 推送流式事件
10. SDK 分发事件给 Mini Bar 和 Skill 小程序

### 5.2 多轮对话时序

1. Skill 小程序调用 `sendMessage`
2. Skill Server 持久化消息并触发 AI
3. WebSocket 推送增量事件
4. SDK 更新缓存并通知监听器
5. 小程序渲染流式文本、工具结果、权限交互

### 5.3 历史恢复与执行中消息恢复时序

1. 用户从 Mini Bar 打开 Skill 小程序，或关闭后重新打开 Skill 小程序
2. IM 客户端将当前 `welinkSessionId` 传给 Skill 小程序
3. SDK 调用 `getSessionMessage`
4. SDK 拉取服务端历史消息
5. SDK 检查本地流式缓存，以及当前会话是否仍处于执行中
6. 若会话处于执行中：
   - 保持当前 WebSocket 监听有效
   - 合并本地尚未落库的流式消息片段
   - 优先使用 `streaming` / `snapshot` 恢复事件补齐进行中的消息状态
7. SDK 对历史消息、恢复事件、本地缓存做统一去重和顺序合并
8. 小程序拿到可直接渲染的消息列表，并继续接收后续流式事件
9. 若应用已经重启，则仅依赖服务端历史消息恢复，不能再使用上次进程内缓存

**执行中恢复原则**：

- SDK 不能只依赖历史消息接口恢复页面
- 必须同时结合活跃 WebSocket 流和本地缓存
- 保证“小程序打开时既能看到已有内容，也能继续渲染后续增量”
- 服务端历史消息是权威来源，本地缓存只补齐运行期未落库内容

### 5.4 选文发送到 IM 时序

1. 用户在小程序中选择目标消息或默认使用当前会话最后一条最终消息
2. 小程序调用 `sendMessageToIM`
3. SDK 从共享缓存中解析最终完整内容
4. SDK 调用 Skill Server 接口
5. Skill Server 调用 IM 平台 API
6. IM 聊天窗口显示文本消息

## 六、关键架构决策

### 6.1 以 SDK 为架构主轴

不把服务端内部实现写入核心架构，避免文档重心偏离客户端集成目标。

### 6.2 REST + WebSocket 双通道模型

REST 负责命令型请求，WebSocket 负责事件型推送。

### 6.3 状态映射由 SDK 内部收敛

上层 UI 不直接消费底层复杂事件，统一由 SDK 输出三态结果。

### 6.4 历史恢复采用“服务端权威 + 本地补齐”

服务端历史消息是主来源，本地缓存只补“尚未落库但已收到”的内容。

### 6.5 打开即接续执行中消息

从 Mini Bar 打开小程序时，不仅要恢复已有消息，还要无缝承接正在执行中的流式消息。

## 七、约束与风险

### 7.1 约束

- 本期仅覆盖 Windows PC、本地验证、OpenCode 单 Skill
- SDK 仅做客户端能力中枢，不负责 IM 宿主 UI 具体实现
- Skill Server 为外部依赖，本架构不展开其内部模块实现

### 7.2 风险

- 多会话复用依赖 `imGroupId + ak + ACTIVE` 查询结果，若服务端返回顺序不稳定，会影响默认复用结果
- 状态映射如果与服务端事件演进不同步，可能导致 UI 状态误判
- 本地缓存与服务端恢复协同时，存在重复或顺序处理复杂度
- IM 客户端与小程序共享 SDK，任何接口调整都可能影响两个接入方
