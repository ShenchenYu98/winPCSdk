## 一、方案概述

### 1.1 项目背景

现有 IM 聊天消息仅支持纯文本，无法直接承载 AI 编码工具的富文本输出，如 Markdown、代码块、流式响应、工具调用结果和权限交互。

本期方案不新建独立 AI 对话窗口，而是通过 IM 已有的 `/` 命令框架触发 Skill 交互，在 IM 聊天窗口与 Skill 小程序之间形成协同体验。Skill SDK 作为客户端统一接入层，负责承接 IM Chat Window、Mini Bar、Skill 小程序与 Skill Server 的通信和状态编排。

### 1.2 方案目标

- 在 IM 内完成 Skill 首次触发、首轮执行和状态展示
- 在 Skill 小程序中承接完整多轮对话能力
- 支撑流式消息、代码块、工具调用、权限请求、问题交互等渲染场景
- 支撑“选文发送到聊天”的业务闭环
- 支撑从 Mini Bar 打开或重新打开小程序时的历史恢复与执行中消息接续
- 保证现有 IM 聊天主流程不受回归影响

### 1.3 本期范围

| 维度 | 范围 |
|------|------|
| 平台 | Windows PC |
| 模式 | 本地验证 |
| 会话 | 个人私有会话 |
| AI 工具 | OpenCode 单 Skill |
| 组件焦点 | Skill SDK 插件 |
| 外部依赖 | IM 客户端、Mini Bar、Skill 小程序、Skill Server、IM 平台 API |

## 二、总体方案

### 2.1 核心思路

Skill SDK 作为客户端交互中枢，对外提供统一接口，对内完成以下工作：

- 管理与 Skill Server 的 REST / WebSocket 通道
- 统一 IM 客户端与 Skill 小程序的调用方式
- 将会话创建/复用、首轮消息发送、多轮对话、状态回调、权限回复、结果发送等能力收敛为标准接口
- 将服务端流式事件统一映射为客户端可消费的 `StreamMessage`
- 在页面打开、重开、从 Mini Bar 展开时，统一处理历史恢复与执行中消息接续

### 2.2 交互节点

| 节点 | 角色 | 职责 |
|------|------|------|
| IM Chat Window | 触发入口 | `/` 触发 Skill、展示高亮 Skill 行、承接首条输入、管理当前会话 ID |
| Mini Bar | 状态承载 | 展示处理中/完成/中止状态，承接展开入口 |
| Skill 小程序 | 主交互界面 | 展示完整对话、继续多轮消息、处理权限与问题、发送结果回 IM |

## 四、关键交互流程

### 4.1 首次触发 Skill

1. 用户在 IM Chat Window 中输入 `/` 并选择 OpenCode Skill。
2. IM Chat Window 展示高亮 Skill 行，用户输入首条问题并点击“生成”。
3. IM Chat Window 调用 SDK `createSession(ak, title, imGroupId)`。
4. Skill SDK 建立或复用 WebSocket 连接。
5. Skill SDK 调用 Skill Server 按 `imGroupId + ak + ACTIVE` 查询可复用会话。
6. 若存在可复用会话，则直接复用；若存在多个，则按服务端返回原始顺序取第一个；若不存在，则创建新会话。
7. Skill SDK 返回 `welinkSessionId` 给 IM 客户端。
8. IM Chat Window 调用 SDK `registerSessionListener(...)` / `onSessionStatusChange(...)` 注册监听。
9. IM Chat Window 调用 SDK `sendMessage(welinkSessionId, content)` 发送首条消息。
10. Skill Server 开始推送首轮流式事件。
11. Skill SDK 将状态映射为 `executing`，通知 Mini Bar 显示“处理中”。
12. Skill SDK 将流式事件分发给 Skill 小程序侧监听器，为后续展开后渲染做准备。

### 4.2 Mini Bar 状态联动

1. Skill Server 持续推送 `step.start`、`session.status`、`text.delta`、`step.done` 等事件。
2. Skill SDK 根据 `type` 和必要附加字段进行状态映射。
3. Skill SDK 通过 `onSessionStatusChange` 向 Mini Bar 透出 `executing / completed / stopped`。
4. Mini Bar 根据状态更新展示内容。
5. 若用户点击“停止”，Mini Bar 调用 SDK `stopSkill(welinkSessionId)`。
6. Skill SDK 调用 Skill Server 中止当前轮执行。
7. Skill SDK 在收到中止结果后向 Mini Bar 回调 `stopped`。

### 4.3 小程序多轮对话

1. 用户在 Skill 小程序中输入新消息。
2. Skill 小程序调用 SDK `sendMessage(welinkSessionId, content)`。
3. Skill SDK 调用 Skill Server 发送用户消息。
4. Skill Server 持续推送本轮流式事件。
5. Skill SDK 将事件分发给 Skill 小程序。
6. Skill 小程序渲染文本、代码块、工具结果、问题与权限交互。
7. 用户可继续发送下一轮消息，重复以上流程。

### 4.4 权限与问题交互

1. Skill Server 推送 `question` 或 `permission.ask`。
2. Skill SDK 将事件分发给 Skill 小程序。
3. Skill 小程序展示提问 UI 或权限确认 UI。
4. 若是提问场景，用户回答后由 Skill 小程序调用 `sendMessage(...toolCallId...)`。
5. 若是权限场景，用户选择同意或拒绝后由 Skill 小程序调用 `replyPermission(...)`。
6. Skill SDK 将用户决策提交给 Skill Server。
7. Skill Server 再继续推送后续流式事件。
8. Skill 小程序继续渲染后续消息和状态变化。

### 4.5 选文发送到 IM

1. 用户在 Skill 小程序中选中目标消息，或默认使用当前会话最后一条最终消息。
2. Skill 小程序调用 SDK `sendMessageToIM(welinkSessionId, messageId?)`。
3. Skill SDK 从共享缓存中解析最终完整内容。
4. Skill SDK 调用 Skill Server 的发送到 IM 接口。
5. Skill Server 调用 IM 平台 API 发送消息。
6. IM 聊天窗口显示该文本消息。
7. 用户在 IM 中看到由 Skill 结果回传的内容。

### 4.6 从 Mini Bar 打开小程序与历史恢复

1. 用户点击 Mini Bar 的“展开”，或重新打开 Skill 小程序。
2. IM 客户端将当前 `welinkSessionId` 传给 Skill 小程序。
3. Skill 小程序初始化时调用 SDK `getSessionMessage(welinkSessionId)`。
4. Skill SDK 向 Skill Server 拉取历史消息。
5. Skill SDK 检查本地是否存在流式缓存。
6. Skill SDK 将历史消息与本地缓存做统一合并。
7. Skill SDK 输出可直接渲染的消息列表给 Skill 小程序。
8. Skill 小程序立即渲染已有消息内容。
9. 若后续仍有流式事件到达，Skill SDK 持续分发给 Skill 小程序。

### 4.7 执行中消息恢复与接续渲染

1. 用户从 Mini Bar 打开 Skill 小程序，或关闭后重新打开 Skill 小程序。
2. IM 客户端将当前 `welinkSessionId` 传给 Skill 小程序。
3. Skill 小程序调用 SDK `getSessionMessage(welinkSessionId)`。
4. Skill SDK 向 Skill Server 拉取已持久化历史消息。
5. Skill SDK 检查当前会话是否仍处于执行中状态。
6. 若会话仍在执行中，Skill SDK 保持当前 WebSocket 监听持续有效。
7. Skill SDK 将本地缓存中的未落库增量消息与历史消息合并。
8. 若 Skill Server 推送 `snapshot` 或 `streaming` 恢复事件，Skill SDK 继续将其纳入合并结果。
9. Skill SDK 对历史消息、恢复事件、本地缓存统一去重和排序。
10. Skill SDK 将合并后的消息列表输出给 Skill 小程序进行首次渲染。
11. 若执行中的消息仍在持续输出，Skill SDK 继续把新的流式增量分发给 Skill 小程序。
12. 小程序监听注册时机由小程序自己控制，SDK 只提供 `registerSessionListener` 能力。
13. 消息列表合并、同一消息增量更新、页面渲染策略由小程序处理，SDK 不负责页面渲染方案。
14. 从 Mini Bar 打开小程序时，执行中的消息可继续渲染且无丢失是小程序与 SDK 协同达成的系统级目标，不是 SDK 单独保证的结果。

## 五、接口能力落地方案

### 5.1 首次触发能力
- 接口：`createSession`、`sendMessage`
- 落地点：IM Chat Window
- 作用：创建或复用会话、建立连接、发送首条消息

### 5.2 状态联动能力
- 接口：`onSessionStatusChange`、`stopSkill`
- 落地点：Mini Bar
- 作用：将流式事件映射为可直接展示的状态

### 5.3 小程序控制能力
- 接口：`controlSkillWeCode`、`onSkillWecodeStatusChange`
- 落地点：Mini Bar、Skill 小程序
- 作用：展开、最小化、关闭联动

### 5.4 流式渲染能力
- 接口：`registerSessionListener`、`unregisterSessionListener`
- 数据结构：`StreamMessage`
- 落地点：Skill 小程序
- 作用：支撑文本、代码块、工具调用、问题、权限、附件等渲染

### 5.5 恢复能力
- 接口：`getSessionMessage`
- 落地点：Mini Bar 打开小程序、重新打开小程序
- 作用：提供历史消息获取、运行期缓存补齐和执行中消息接续能力；页面渲染与消息合并由小程序负责

### 5.6 回传能力
- 接口：`sendMessageToIM`
- 落地点：Skill 小程序
- 作用：按会话/消息标识解析最终内容并发送到 IM

### 5.7 权限交互能力
- 接口：`replyPermission`
- 落地点：Skill 小程序
- 作用：承接 AI 请求的权限确认闭环

## 六、状态映射落地规则

### 6.1 映射来源

状态映射基于流式消息的 `type`，必要时读取附加字段 `sessionStatus`、`response`。

### 6.2 映射规则

| 消息类型 | 条件 | 状态 |
|------|------|------|
| `step.start` | 无 | `executing` |
| `session.status` | `busy` / `retry` | `executing` |
| `text.delta` | 无 | `executing` |
| `thinking.delta` | 无 | `executing` |
| `tool.update` | 无 | `executing` |
| `question` | 无 | `executing` |
| `permission.ask` | 无 | `executing` |
| `permission.reply` | `once` / `always` | `executing` |
| `file` | 无 | `executing` |
| `step.done` | 无 | `completed` |
| `session.status` | `idle` | `completed` |
| `text.done` | 无 | `completed` |
| `thinking.done` | 无 | `completed` |
| `permission.reply` | `reject` | `stopped` |
| `session.error` | 无 | `stopped` |
| `error` | 无 | `stopped` |
| `agent.offline` | 无 | `stopped` |

### 6.3 特殊说明

- `session.title` 不参与状态映射
- `text.done`、`thinking.done` 是部件级完成信号
- 更稳定的整轮完成信号优先看 `step.done` 或 `session.status=idle`
- `stopSkill()` 成功后，SDK 额外触发一次 `stopped`

## 七、方案约束

- 方案文档聚焦 SDK 落地，不展开 Skill Server 内部设计
- 方案按业务交互主线组织，不替代架构设计文档
- 本期不设计多工具抽象能力
- 本期默认服务端已具备 SDK 所需接口能力

---

## 八、方案验收要点

- FR-1 到 FR-9 均能在交互节点上找到对应 SDK 接口承接点
- 从 IM 触发到小程序多轮对话链路完整
- Mini Bar 状态变化可解释、可回溯
- 从 Mini Bar 打开小程序时，执行中的消息可继续渲染且无丢失
- 选文发送到 IM 能形成业务闭环
- 会话创建/复用、首条消息发送、多轮消息发送三者职责边界清晰
