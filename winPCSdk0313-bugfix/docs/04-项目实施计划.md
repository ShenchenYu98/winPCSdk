# Skill SDK 项目实施计划

**版本**: v1.0  
**日期**: 2026-03-08  
**状态**: 待确认  
**项目代号**: Skill SDK

---

## 一、项目概述

### 1.1 项目目标

围绕 Skill SDK 插件完成本期需求交付，支撑以下业务闭环：

- IM Chat Window 中首次触发 Skill
- Mini Bar 状态联动
- Skill 小程序多轮对话
- 流式消息与富文本事件支撑
- 历史消息恢复与执行中消息接续
- 选文发送到 IM
- 权限确认与生命周期控制
- 基于参考 OpenCode CUI 页面形态，提供一个测试用对话聊天页面，用于覆盖 Skill SDK 全业务链路场景验证
- 通过本地 Mock Skill Server 能力，在无真实服务端依赖下完成主链路、恢复链路和交互链路测试

### 1.2 项目范围

本期仅覆盖：

- Windows PC
- 本地验证
- 个人私有会话
- OpenCode 单 Skill
- Skill SDK 客户端能力交付
- 测试用聊天页面与本地 Mock 服务端

本期不覆盖：

- 多工具支持
- 远程模式
- 团队共享
- 生产级高可用部署
- Skill Server 内部实现改造计划
- 真实 IM 宿主页面的完整产品化改造计划

### 1.3 交付原则

- 以 SDK 能力闭环为主，不扩展到宿主 UI 具体实现细节
- 以阶段/里程碑组织实施，但不做周级排期承诺
- 每阶段都要求有可验证交付物
- 优先打通主链路，再补足恢复、状态、回传等增强能力
- 每个接口交付时必须同步具备对应单元测试
- 在接口单元测试之外，必须补充真实场景模拟测试

## 三、分阶段实施计划

### 阶段一：主链路打通（M1）

#### 目标
完成 Skill SDK 最小可运行能力，使 IM 入口能够触发首轮 Skill 对话。

#### 任务分组

**任务组 A：SDK 基础骨架**
- 建立 SDK 对外 Facade 结构
- 明确会话、监听器、缓存、状态映射的内部模块边界
- 统一 IM Chat Window 与 Skill 小程序的调用入口

**任务组 B：会话创建与复用链路**
- 实现 `createSession` 的完整链路：建连、按 `imGroupId + ak + ACTIVE` 查会话、复用或新建会话
- 明确多会话命中时按服务端返回原始顺序取第一个
- 编写 `createSession` 单元测试
- 实现会话标识和监听器按 `welinkSessionId` 路由

**任务组 C：首轮消息发送与流式接收**
- 实现 `sendMessage`
- 编写 `sendMessage` 单元测试
- 实现 `registerSessionListener`
- 编写 `registerSessionListener` 单元测试
- 打通首轮从 IM Chat Window 到 Skill Server 再回到客户端的链路

**任务组 D：连接关闭与监听器移除能力**
- 实现 `closeSkill`
- 编写 `closeSkill` 单元测试
- 实现 `unregisterSessionListener`
- 编写 `unregisterSessionListener` 单元测试
- 明确 `closeSkill` 和 `unregisterSessionListener` 的幂等行为

#### 先后顺序
1. SDK 基础骨架
2. 会话创建与复用链路
3. 首轮消息发送与流式接收
4. 连接关闭与监听器移除能力
5. 主链路联调

#### 交付物
- Skill SDK 基础目录和模块边界
- `createSession`、`sendMessage`
- `registerSessionListener`、`unregisterSessionListener`、`closeSkill`
- 对应接口单元测试用例与执行结果
- 首轮消息联调记录

#### 里程碑验收
- 用户点击“生成”后可成功创建或复用 Skill 会话
- 首条消息可触发 AI 返回
- 流式事件可被 SDK 收到并分发
- `closeSkill` 与 `unregisterSessionListener` 满足幂等要求
- 本阶段涉及接口均具备单元测试，覆盖成功路径与关键失败路径

### 阶段三：恢复与结果回传（M3）

#### 目标
完成历史恢复、执行中消息接续和结果发送到 IM 的闭环能力。

#### 任务分组

**任务组 A：历史消息恢复**
- 实现 `getSessionMessage`
- 编写 `getSessionMessage` 单元测试
- 拉取服务端历史消息
- 统一消息分页、顺序、去重输出
- 明确服务端历史消息是权威来源

**任务组 B：执行中消息接续**
- 建立本地流式缓存管理
- 接入 `snapshot` / `streaming` 恢复事件
- 处理从 Mini Bar 打开小程序时的执行中消息接续
- 保证“已有内容可见 + 后续增量不断流”
- 明确 `closeSkill` 后缓存保留、小程序关闭后缓存保留、应用重启后缓存不可用

**任务组 C：选文发送到 IM**
- 实现 `sendMessageToIM`
- 编写 `sendMessageToIM` 单元测试
- 基于 `welinkSessionId` / `messageId` 解析最终消息内容
- 打通 Skill 小程序到 IM 的结果回传链路

**任务组 D：权限交互闭环**
- 实现 `replyPermission`
- 编写 `replyPermission` 单元测试
- 支撑 `permission.ask` / `permission.reply` 的前后链路闭环

#### 先后顺序
1. 历史消息恢复
2. 执行中消息接续
3. 选文发送到 IM
4. 权限交互闭环

## 七、单元测试要求

### 7.1 接口级测试范围

| 接口 | 最少测试方向 |
|------|-------------|
| `createSession` | 会话复用、新建会话、按 `imGroupId + ak + ACTIVE` 查询、多个会话命中时取第一个、建连失败、查会话失败 |
| `closeSkill` | 正常关闭、重复关闭、未连接关闭、关闭后资源清理 |
| `stopSkill` | 正常停止、停止失败、停止后状态回调 |
| `onSessionStatusChange` | 各类 `type` 映射、附加字段映射、状态切换边界 |
| `onSkillWecodeStatusChange` | close/minimize 回调触发 |
| `regenerateAnswer` | 找到最后一条用户消息、优先复用本地缓存、无可重发消息 |
| `sendMessageToIM` | 根据消息标识获取最终内容、无 `messageId` 使用最后一条消息、发送失败 |
| `getSessionMessage` | 历史消息分页、缓存合并、执行中恢复、去重排序、应用重启后无本地缓存 |
| `registerSessionListener` | 注册成功、先注册后建连、缓存注册信息、多监听器分发 |
| `unregisterSessionListener` | 正常移除、重复移除、监听器不存在、移除后不再接收消息 |
| `sendMessage` | 首次发送、后续多轮、带 `toolCallId`、发送失败、自动建连兜底 |
| `replyPermission` | 同意一次、始终同意、拒绝、接口失败 |
| `controlSkillWeCode` | close、minimize、异常分支 |

### 7.2 覆盖要求

每个接口至少覆盖以下三类测试：

- **成功路径**：标准调用能够得到预期结果
- **失败路径**：服务端失败、参数不合法、连接异常等场景
- **边界场景**：时序、状态切换、重复调用、执行中恢复等关键边界

### 7.3 收口标准

- 所有对外 SDK 接口均有对应单元测试
- 单元测试结果可追溯到接口名和场景名
- 测试覆盖清单能支撑阶段验收和最终文档交付

### 7.4 真实场景模拟测试要求

真实场景模拟测试至少覆盖以下场景：

- 首次触发 Skill 并完成 `createSession + sendMessage`
- Mini Bar 状态从 `executing` 到 `completed / stopped`
- 小程序中继续多轮对话
- `question` 与 `permission.ask` 交互闭环
- `sendMessageToIM` 的结果回传
- 从 Mini Bar 打开小程序时的历史恢复
- 执行中状态下打开小程序并继续流式渲染
- `snapshot` / `streaming` 恢复事件的去重与续接
- 小程序关闭、最小化、重新打开的生命周期联动
- 本地 Mock 服务端下的异常注入和失败路径验证

## 八、实施约束与风险

### 8.1 实施约束
- 当前计划仅针对 Skill SDK，不拆分服务端内部任务
- 不做周级排期承诺
- 外部依赖能力默认可用，但联调时仍需验证接口一致性
- 真实场景模拟测试阶段以本地 Mock 实现为准，不等同于真实生产环境行为

### 8.2 实施风险
- 多会话复用依赖 `imGroupId + ak + ACTIVE` 查询结果，若服务端返回顺序不稳定，可能影响默认复用结果
- 状态映射与服务端事件演进不一致，可能导致 UI 展示偏差
- 执行中恢复是实现复杂度最高的部分，容易出现丢消息或重复消息
- IM Chat Window、Mini Bar、Skill 小程序三节点协作对接口时序要求高
- 若单元测试未及时跟上接口变更，后续联调阶段会放大问题排查成本
- 测试页面若与真实接入页面交互模型差异过大，可能降低场景测试价值
- Mock 服务端事件行为若与真实 Skill Server 偏差过大，可能导致测试结果失真
