**版本**: v1.0  
**日期**: 2026-03-08  
**状态**: 待确认  
**项目代号**: Skill SDK

---

## 一、产品概述

### 1.1 产品定位

Skill SDK 是集成在 IM 客户端侧的一层 SDK 插件，用于承接 IM 聊天窗口、Skill 小程序与 Skill Server 之间的交互能力，使用户能够在现有 IM 工具内完成 AI 对话触发、状态展示、多轮会话、流式渲染协同与结果回传。

### 1.2 目标用户

- **主要用户**: 使用现有企业 IM 工具进行开发协作的开发者
- **次要用户**: 负责客户端集成的 IM 前端/客户端研发人员
- **协同用户**: Skill 小程序研发、Skill Server 研发、测试人员

### 1.3 核心价值

1. **集成化使用**: 用户无需离开 IM，即可触发并使用 AI Skill 能力。
2. **统一交互协议**: IM 客户端和 Skill 小程序通过同一套 SDK 能力接入 Skill Server。
3. **流式体验支撑**: 支持流式消息、状态同步、权限交互、工具结果透出。
4. **可恢复会话体验**: 支持重新打开 Skill 小程序后恢复历史消息与会话状态。
5. **低侵入接入**: 尽量不改变现有 IM 聊天主流程，降低集成改造成本。

## 二、需求范围

### 2.1 本期范围

本期需求仅覆盖以下范围：

- **平台范围**: Windows PC
- **技术形态**: Electron + React + TypeScript 客户端集成
- **验证方式**: 本地验证
- **会话模式**: 个人私有会话
- **AI 工具范围**: OpenCode 单 Skill
- **组件范围**: Skill SDK 插件能力定义，不包含 IM 既有能力改造实现细节

### 2.2 非范围

以下内容不纳入本期需求：

- Claude Code / OpenClaw / 其他 AI 工具支持
- 远程模式
- 团队共享会话
- 独立 AI 对话窗口产品
- IM 服务端能力改造方案
- Skill Server 的详细服务端实现设计
- 生产级监控、运维、高可用集群方案

### 2.3 外部依赖与系统边界

#### 我们构建的组件

| 组件 | 性质 | 运行位置 | 职责 |
|------|------|----------|------|
| **Skill SDK 插件** | SDK 插件 | IM 客户端 | 承接 IM 客户端、Skill 小程序与 Skill Server 之间的交互能力 |

#### 外部已有组件

- IM 聊天客户端
- IM 服务端 / 平台 API

## 三、业务目标

### 3.1 业务目标

- 支撑 IM 内通过 "/" 命令触发 OpenCode Skill
- 支撑首轮问题提交后的 Mini Bar 状态联动
- 支撑 Skill 小程序展开后的完整多轮对话
- 支撑流式富文本输出与工具交互类事件透出
- 支撑用户将 AI 返回内容发送回 IM 聊天
- 支撑会话历史恢复，避免重复发起新会话
- 保证现有 IM 聊天主流程不受回归影响

### 3.2 成功标准

- 用户可在 IM 内完成从 Skill 触发到多轮对话的完整闭环
- SDK 能覆盖 Skill 小程序与 IM 客户端所需的主要交互接口
- SDK 接口能力可直接映射主文档 AC-1 至 AC-7
- SDK 对接 Skill Server 的事件流和状态流具备可解释性与可维护性

## 四、用户场景

### 4.1 场景一：IM 中首次触发 Skill

用户在 IM 输入框中输入 `/`，选择 OpenCode Skill。聊天框中出现高亮 Skill 名称，用户输入首条问题并点击“生成”。SDK 先创建或复用会话并建立连接，再发送首条消息触发首轮 AI 执行，并驱动 Mini Bar 进入处理中状态。

### 4.2 场景二：展开 Skill 小程序继续多轮对话

用户点击 Mini Bar 的“展开”，进入 Skill 小程序。SDK 持续接收 Skill Server 的流式事件，驱动小程序渲染文本、代码块、工具调用和提问交互。用户可继续发送多轮消息，保持在同一会话中上下文连续。

### 4.3 场景三：选中回复内容发送到 IM

用户在 Skill 小程序中选中 AI 返回的某段文本，点击“发送到聊天”。SDK 将用户最终确认的文本发送到 IM 聊天窗口，形成从 Skill 对话到 IM 消息的结果回传闭环。

### 4.4 场景四：关闭后重新打开恢复历史

用户关闭或收起 Skill 小程序，稍后再次展开。SDK 拉取并合并历史消息与必要的流式恢复状态，使用户能够继续查看和使用之前的对话内容，而不是重新开始新会话。运行期可结合本地缓存补齐未落库或进行中的消息内容；应用重启后的恢复以服务端历史消息为准。

## 五、功能需求

### FR-1: Skill 首次触发与会话创建
**优先级**: P0（必须）

**描述**: IM 客户端在用户选择 Skill 并输入首条问题后，通过 SDK 先创建或复用会话，再发送首条消息触发首轮 AI 执行。

**功能点**:
- 提供创建或复用会话的统一入口能力
- 支持按 `imGroupId + ak` 查询和复用活跃会话
- 支持建立或复用与 Skill Server 的 WebSocket 长连接
- 支持发送首条用户消息内容
- 首轮 AI 响应通过流式事件回传

**对应 SDK 能力**:
- `createSession`
- `sendMessage`

**验收标准**:
- 用户首次点击“生成”后能成功进入 Skill 会话
- 首条消息成功触发 AI 执行
- 后续流式消息能被监听器接收

---

### FR-2: Mini Bar 状态协同
**优先级**: P0（必须）

**描述**: IM 客户端需要通过 SDK 感知 Skill 会话状态变化，并驱动 Mini Bar 展示“处理中 / 完成 / 中断”等状态。

**功能点**:
- 提供会话状态变化回调
- 允许基于流式事件 `type` 和必要附加字段进行状态映射
- 支持处理中、完成、中止三态透出
- 支持停止当前轮后回调状态变更

**对应 SDK 能力**:
- `onSessionStatusChange`
- `stopSkill`

**验收标准**:
- 首轮执行开始后 Mini Bar 可显示处理中
- 当前轮完成后 Mini Bar 可更新为完成
- 用户主动停止后状态可更新为中止

---

### FR-3: Skill 小程序生命周期协同
**优先级**: P0（必须）

**描述**: Skill 小程序展开、最小化、关闭时，SDK 需要向上层提供统一控制与状态回调能力。

**功能点**:
- 支持小程序关闭
- 支持小程序最小化
- 支持状态变化回调通知 IM 客户端
- 保持小程序状态与 IM 聊天主界面切换一致

**对应 SDK 能力**:
- `controlSkillWeCode`
- `onSkillWecodeStatusChange`

**验收标准**:
- 点击展开后可进入 Skill 小程序
- 点击收起后可回到 IM 聊天界面
- 最小化和关闭均能触发对应状态回调

---

### FR-4: 多轮对话能力
**优先级**: P0（必须）

**描述**: 用户在 Skill 小程序中可继续发送消息，与当前会话进行多轮对话。

**功能点**:
- 支持发送后续用户消息
- 支持在当前会话下持续接收 AI 流式回复
- 支持中断后继续发送新消息
- 支持重新生成上一轮回答

**对应 SDK 能力**:
- `sendMessage`
- `stopSkill`
- `regenerateAnswer`

**验收标准**:
- 小程序内可以持续进行多轮消息交互
- 中断当前轮后不影响下一轮继续对话
- 重新生成可基于当前会话上下文触发

---

### FR-5: 流式事件监听与富文本渲染支撑
**优先级**: P0（必须）

**描述**: SDK 需要向 Skill 小程序提供完整的事件流能力，以支撑流式文本、思维链、工具调用、提问、权限、附件等 UI 渲染。

**功能点**:
- 提供统一会话监听器注册与移除能力
- 对齐 Skill Server 的 WebSocket 事件模型
- 支持文本增量、文本完成、思维链、工具调用、问题交互、权限请求、附件、会话状态、快照恢复等事件
- 支持监听器先注册、连接后建立的场景

**对应 SDK 能力**:
- `registerSessionListener`
- `unregisterSessionListener`
- `StreamMessage`

**验收标准**:
- 流式消息可逐步显示
- 代码块、Markdown、Tool Use 可获得必要事件支撑
- 权限与提问事件可被正确识别并驱动 UI

---

### FR-6: 历史消息恢复
**优先级**: P0（必须）

**描述**: 用户重新打开 Skill 小程序后，可恢复当前会话历史消息，并合并尚未落库的流式缓存。

**功能点**:
- 支持拉取会话历史消息列表
- 支持分页获取
- 支持将流式缓存与历史消息合并
- 支持断线恢复后的快照和进行中消息恢复
- 服务端历史消息作为权威来源，本地缓存仅用于运行期补齐未落库或进行中的消息

**对应 SDK 能力**:
- `getSessionMessage`

**验收标准**:
- 重新打开小程序后可看到历史消息
- 正在进行中的消息不会因重开而丢失
- 消息顺序正确、无明显重复
- 应用重启后的恢复以服务端历史消息为准

---

### FR-7: 选文发送到 IM
**优先级**: P0（必须）

**描述**: 用户可在 Skill 小程序中选择最终确认的文本内容，通过 SDK 发送回 IM 聊天。

**功能点**:
- 提供发送到 IM 的统一接口
- 接收最终确认的文本内容，而非选区坐标
- 关联当前会话上下文
- 支撑“选文发送到聊天”的业务闭环

**对应 SDK 能力**:
- `sendMessageToIM`

**验收标准**:
- 用户可将 AI 返回的任意选中文本发送到 IM
- IM 聊天区可看到该文本消息
- SDK 接口语义明确，可被小程序直接调用

---

### FR-8: 权限确认交互
**优先级**: P1（重要）

**描述**: 当 AI 请求执行敏感操作时，SDK 需要支持权限请求展示与用户回复。

**功能点**:
- 支持接收权限请求事件
- 支持同意一次、始终同意、拒绝三种响应
- 支持将用户的权限决策回传服务端
- 支持权限请求对当前轮状态的影响透出

**对应 SDK 能力**:
- `replyPermission`
- `permission.ask` / `permission.reply` 事件

**验收标准**:
- 权限请求信息可被前端展示
- 用户可完成批准或拒绝操作
- 回复结果可正确回传服务端

---

### FR-9: 连接与资源释放
**优先级**: P1（重要）

**描述**: SDK 需要支持主动关闭 WebSocket 连接和释放本地连接资源。

**功能点**:
- 支持关闭 WebSocket 连接
- 支持清理监听器和本地连接态
- 支持窗口关闭后的资源回收
- 保留本地缓存用于运行期会话恢复

**对应 SDK 能力**:
- `closeSkill`

**验收标准**:
- SDK 主动关闭后不再接收新的流式消息
- 本地连接态被释放
- 本地缓存保留用于会话恢复
- 不影响上层对服务端会话的后续处理

---

## 六、非功能需求

### NFR-1: 性能要求
- 流式消息到达客户端后的 UI 感知延迟应尽可能低
- 普通消息历史恢复在常见会话规模下应保持可接受速度
- 状态回调不应阻塞主消息流处理

### NFR-2: 稳定性要求
- SDK 在监听器先注册、连接后建立的时序下应保持行为稳定
- 小程序关闭、最小化、重新展开后不应出现明显状态错乱
- 同一会话的流式消息应尽量避免丢失、乱序、重复渲染

### NFR-3: 兼容性要求
- 兼容 Windows PC 客户端运行环境
- 兼容 Electron + React + TypeScript 集成方式
- 兼容 IM 客户端与 Skill 小程序共用同一套 SDK 能力

### NFR-4: 可维护性要求
- SDK 对外接口语义清晰
- 状态映射规则可解释、可回溯
- 事件模型与服务端协议尽量保持一致
- 文档和接口定义应能被客户端研发直接使用

### NFR-5: 回归控制要求
- 不影响现有 IM 聊天基础功能
- Skill 能力接入应尽量低侵入
- 小程序相关逻辑不应污染普通聊天通道

## 七、验收标准

本期需求验收直接对齐需求概述中的 AC-1 至 AC-7：

- **AC-1**: 用户在 IM 输入框输入 "/" 后展示 OpenCode Skill，点击后聊天框显示高亮 Skill 名称并可输入问题点击“生成”
- **AC-2**: 点击“生成”后 IM 聊天区出现 1 行高的 Skill Mini Bar，显示“OpenCode 处理中…”
- **AC-3**: 点击 Mini Bar 的“展开”后，Skill 小程序覆盖 IM 聊天窗，可继续多轮对话
- **AC-4**: AI 响应以流式方式逐字显示，代码块支持语法高亮和复制按钮
- **AC-5**: 用户可选中 OpenCode 返回的任意文本并发送到聊天
- **AC-6**: 重新打开 Skill 小程序后，历史会话和消息从数据库恢复
- **AC-7**: 现有 IM 聊天功能不受影响，无明显回归

## 八、风险与约束

### 8.1 已知约束

- 本期仅定义客户端 SDK 需求，不覆盖服务端完整实现方案
- 本期为本地验证，不代表生产部署形态
- 本期只考虑 OpenCode 单 Skill，不设计多工具抽象层需求

### 8.2 已知风险

- `closeSkill` 当前只负责关闭 WebSocket，本身不等同于关闭服务端会话
- 会话恢复能力依赖服务端持久化和流式恢复协同，实际体验受服务端实现完整度影响
- IM 客户端与 Skill 小程序共用同一套 SDK 能力，对状态同步和时序一致性要求较高

### 8.3 依赖条件

- IM 客户端已有 "/" 命令框架和 Mini Bar UI 壳能力
- Skill 小程序具备接入 SDK 的运行容器
- Skill Server 提供 REST + WebSocket 能力
- IM 平台 API 已存在文本消息发送能力
