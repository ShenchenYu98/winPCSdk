# Skill Client SDK 方案设计文档

## 1. 文档信息
- 文档版本：v1.0（评审稿）
- 事实来源：`documents/skillSdk/SkillClientSdkInterface.md`
- 决策基线：PRD 与架构文档已评审结论（2026-03-07）
- 产出日期：2026-03-07

## 2. 方案目标与实施约束
## 2.1 目标
- 将 13 个既有接口落为可开发、可联调、可测试的实现方案。
- 统一会话上下文、连接策略、监听器分发、历史消息合并规则。
- 在不新增业务接口前提下，完成 SDK MVP 交付。

## 2.2 约束（已定）
1. `closeSkill` 无入参，作用域为“SDK 管理的全部会话连接”。
2. `stopSkill` 语义目标为“停止本轮生成”；服务端仍使用 `DELETE /sessions/{id}`，本期暂不改造。
3. `SkillSession.id` 统一为 `string`。
4. `userId` 在 SDK 统一为 `string`。
5. `onSessionStatusChange` 在无 WS 连接时不允许自动建连。
6. 统一错误对象结构。
7. `StreamMessage.sessionId` 必填。

## 3. 公共契约章节（四文档统一）
## 3.1 接口能力矩阵
| 分组 | 接口 | 输入 | 输出 | 依赖 | 核心副作用 |
|---|---|---|---|---|---|
| 会话执行 | executeSkill | ExecuteSkillParams | SkillSession | REST + WS | 创建会话并触发首轮消息 |
|  | sendMessage | SendMessageParams | SendMessageResult | REST + WS | 触发多轮对话 |
|  | regenerateAnswer | RegenerateAnswerParams | AnswerResult | REST + WS | 复用最后一条用户消息重答 |
|  | stopSkill | StopSkillParams | StopSkillResult | REST | 中断本轮生成 |
|  | closeSkill | 无 | CloseSkillResult | WS | 关闭所有受管连接 |
| 回调监听 | registerSessionListener | RegisterSessionListenerParams | void | WS | 注册回调并接收流 |
|  | unregisterSessionListener | UnregisterSessionListenerParams | void | WS | 移除回调 |
|  | onSessionStatusChange | OnSessionStatusChangeParams | void | WS 既有连接 | 执行态回调 |
|  | onSkillWecodeStatusChange | OnSkillWecodeStatusChangeParams | void | 本地事件 | 小程序状态回调 |
| 消息与协同 | getSessionMessage | GetSessionMessageParams | PageResult\<ChatMessage\> | REST + Cache | 历史+流式合并 |
|  | sendMessageToIM | SendMessageToIMParams | SendMessageToIMResult | REST | 转发 IM |
|  | replyPermission | ReplyPermissionParams | ReplyPermissionResult | REST | 权限确认 |
|  | controlSkillWeCode | ControlSkillWeCodeParams | ControlSkillWeCodeResult | 宿主能力 | 控制小程序 |

## 3.2 会话状态机映射
- 生命周期：`ACTIVE / IDLE / CLOSED`
- 执行态：`executing / stopped / completed`
- 映射规则：
  - `delta` -> `executing`
  - `done` -> `completed`
  - `error/agent_offline` -> `stopped`
  - `agent_online` -> `executing`

## 3.3 流式消息协议
```typescript
interface StreamMessage {
  sessionId: string; // 必填（已决策）
  type: 'delta' | 'done' | 'error' | 'agent_offline' | 'agent_online';
  seq: number;
  content: string;
  usage?: { inputTokens?: number; outputTokens?: number };
}
```

## 3.4 一致性问题落地策略
1. `closeSkill` 与 `CloseSkillParams` 冲突：以无入参 API 为准，移除对 `CloseSkillParams` 的运行时依赖。
2. `stopSkill` 与 `DELETE` 冲突：MVP 保留 `DELETE` 调用，SDK 通过错误映射兜底用户体验。
3. `SkillSession.sessionId/id` 冲突：SDK 对外统一 `id`。
4. `userId` 类型冲突：SDK 输入输出统一为 `string`，适配层做类型转换。
5. 状态回调建连冲突：`onSessionStatusChange` 仅读取既有连接事件。

## 4. 模块级设计与职责分配
## 4.1 API Facade（SkillClient）
```typescript
interface SkillClient {
  executeSkill(params: ExecuteSkillParams): Promise<SkillSession>;
  closeSkill(): Promise<CloseSkillResult>;
  stopSkill(params: StopSkillParams): Promise<StopSkillResult>;
  onSessionStatusChange(params: OnSessionStatusChangeParams): void;
  onSkillWecodeStatusChange(params: OnSkillWecodeStatusChangeParams): void;
  regenerateAnswer(params: RegenerateAnswerParams): Promise<AnswerResult>;
  sendMessageToIM(params: SendMessageToIMParams): Promise<SendMessageToIMResult>;
  getSessionMessage(params: GetSessionMessageParams): Promise<PageResult<ChatMessage>>;
  registerSessionListener(params: RegisterSessionListenerParams): void;
  unregisterSessionListener(params: UnregisterSessionListenerParams): void;
  sendMessage(params: SendMessageParams): Promise<SendMessageResult>;
  replyPermission(params: ReplyPermissionParams): Promise<ReplyPermissionResult>;
  controlSkillWeCode(params: ControlSkillWeCodeParams): Promise<ControlSkillWeCodeResult>;
}
```

## 4.2 SessionStore
- 数据结构：`Map<sessionId, SessionContext>`
- 字段：`id/userId/lifecycle/executionStatus/connectionState/lastSeq/lastActiveAt`
- 职责：
  - 管理会话生命周期与执行态。
  - 对 `closeSkill` 执行“全部会话连接”遍历关闭。
  - `closeSkill` 成功后清空全部会话内存上下文。

## 4.3 ConnectionManager
- 数据结构：`Map<sessionId, WebSocketClient>`
- 职责：
  - 管理 WS 建连、重连、心跳。
  - 连接事件上报 ListenerBus。
  - `closeSkill` 时统一关闭全部连接。

## 4.4 ListenerRegistry + ListenerBus（含分发隔离）
- 数据结构：
  - `Map<sessionId, ListenerBucket>`
  - `ListenerBucket = {message, error, close, status}`
  - `Map<listenerId, FaultCounter>`
- 关键策略（已采纳）：
  - 逐监听器 `try-catch` 隔离。
  - 连续异常达到阈值（默认 5）触发熔断，暂停该监听器分发并上报。
  - 熔断自动恢复本期不处理，仅支持调用方手动重新注册恢复。

## 4.5 MessageMergeEngine
- 输入：
  - 服务端历史消息（分页）
  - 本地流式缓存（`streaming-{sessionId}`）
- 去重键：
  - 已落库消息：`id`
  - 未落库流式：`sessionId + seq + isStreaming`
- 输出：可渲染列表（历史 + 流式中间态）

## 4.6 ErrorNormalizer
- 输入：HTTP 错误、WS 错误、业务异常
- 输出统一对象：
```typescript
interface SkillSdkError {
  code: string;
  message: string;
  httpStatus?: number;
  retriable: boolean;
  source: 'REST' | 'WS' | 'SDK';
  sessionId?: string;
  timestamp: number;
}
```

## 5. REST/WS 交互细节
## 5.1 REST 端点映射
| 能力 | 方法 | 端点 |
|---|---|---|
| 创建会话 | POST | `/api/skill/sessions` |
| 发送消息/重答 | POST | `/api/skill/sessions/{sessionId}/messages` |
| 停止会话（现网） | DELETE | `/api/skill/sessions/{sessionId}` |
| 获取消息分页 | GET | `/api/skill/sessions/{sessionId}/messages?page={page}&size={size}` |
| 发送到 IM | POST | `/api/skill/sessions/{sessionId}/send-to-im` |
| 权限确认回复 | POST | `/api/skill/sessions/{sessionId}/permissions/{permissionId}` |

## 5.2 WS 连接与路由
- 端点：`ws://{host}:8082/ws/skill/stream`
- 路由规则：
  - 以 `message.sessionId` 作为一级路由键。
  - 同一 session 内按 `seq` 做顺序性检查。
  - `onSessionStatusChange` 仅消费 ListenerBus 已分发事件，不触发建连。

## 5.3 `stopSkill` 现网兼容处理（DELETE 暂不改造）
1. SDK 调用 `DELETE /sessions/{id}`。
2. 本地先发出 `executionStatus=stopped` 回调，停止本轮分发。
3. 若后续 `sendMessage` 返回 404/409，则标准化为：
   - `code=SESSION_TERMINATED_AFTER_STOP`
   - `retriable=true`
   - SDK 仅返回标准错误语义，具体交互（如“重新开始”按钮）由前端控制。
4. 在监控中单独统计该路径命中率，作为服务端后续改造依据。

## 6. 消息合并去重策略
## 6.1 合并流程
1. 拉取服务端分页历史。
2. 读取本地 `StreamAccumulator(sessionId)`。
3. 判断历史中是否已包含同轮最终消息。
4. 未包含则追加临时流式消息，标记 `meta.isStreaming=true`。
5. 返回合并后 `PageResult`。

## 6.2 去重规则
- 规则 A：若历史存在同 `id`，丢弃本地同条缓存。
- 规则 B：若历史最后一条为 ASSISTANT 且内容前缀覆盖流式缓存，优先历史。
- 规则 C：`done` 到达后若尚未落库，仍保留短暂本地项，直到下一次分页确认。

## 7. 时序安全机制
## 7.1 监听先注册后执行
- `registerSessionListener` 可在 `executeSkill` 前调用。
- SDK 暂存监听器并在建连后立即生效。

## 7.2 执行先发后注册
- 对于已连接会话，后注册监听器仅接收后续消息，不回放历史 delta。
- 需要历史内容时，调用 `getSessionMessage` 补齐。

## 7.3 断线恢复
- 重连成功后检查 `seq` 连续性。
- 存在缺口触发历史补拉 + 本地缓存重建。

## 8. 配置与开关
## 8.1 ConnectionPolicy（已决策）
```typescript
interface ConnectionPolicy {
  maxRetryCount: 5;             // 最大重试次数
  backoffInitialMs: 1000;       // 退避起始间隔 1s
  backoffMaxMs: 5000;           // 最大退避间隔 5s
  heartbeatIntervalMs: 15000;   // 心跳间隔 15s
  disconnectThresholdMs: 30000; // 判定断线阈值 30s
}
```

初始化约束：
- `ConnectionPolicy` 通过 SDK 初始化参数透出。
- 支持按环境覆写（dev/test/prod）。
- 未传入时使用默认值。

## 8.2 其他开关
- `autoConnectOnRegister`: `true`（仅对 `registerSessionListener` 生效）。
- `autoDisconnectWhenNoListeners`: `true`（可配置）。
- `defaultPage`: `0`。
- `defaultSize`: `50`。
- `listenerCircuitBreakerThreshold`: `5`（连续异常熔断阈值）。

## 9. 联调方案与灰度策略
## 9.1 联调分层
1. SDK 本地模拟联调：Mock REST/WS，验证状态机和回调分发。
2. 与 Skill Server 联调：重点验证 `sessionId` 路由、`seq` 连续性、stop/delete 兼容路径。
3. IM 与小程序联调：验证消息投递、窗口状态事件闭环。

## 9.2 灰度策略
- Phase A：内部开发环境全量验证。
- Phase B：测试环境 20% 会话灰度（按 userId 哈希）。
- Phase C：预发布环境全量，观察 48 小时核心指标。
- 回滚条件：接口成功率 < 99.5% 或回调丢失率 > 0.3%。
- 监控新增：`dispatchLatencyMs`（WS 收包到监听器回调完成耗时）。

## 10. 方案验收测试（与 PRD 对齐）
1. 首次执行技能到流式完成闭环。
2. 监听先注册/后注册两种时序不丢消息。
3. 多轮对话连续发送与并发回调分发。
4. `stopSkill` 后继续对话行为校验（含 delete 兼容分支）。
5. `closeSkill` 后资源释放与重复关闭处理。
6. 分页历史与流式缓存合并去重。
7. 权限确认批准/拒绝双路径。
8. `sendMessageToIM` 成功与典型失败码。
9. 小程序 `close/minimize` 控制与状态回调。
10. 断线重连与状态恢复一致性。
11. 错误码映射与前端提示一致性。
12. 向后兼容：旧调用方最小改动可用。

## 11. 评审结论（已定稿）
1. `closeSkill` 关闭全部会话连接后，同步清空全部 SessionStore 内存上下文。
2. `SESSION_TERMINATED_AFTER_STOP` 的交互由前端控制，SDK 不处理具体 UI 行为。
3. Listener 熔断自动恢复本期不处理。

## 12. MVP 采纳项（已定稿）
1. `ConnectionPolicy` 透出为 SDK 初始化参数，支持环境覆写（dev/test/prod）。
2. 增加 `dispatchLatencyMs` 指标。
3. `getSessionMessage(includeStreaming=false)` 本期不做。
