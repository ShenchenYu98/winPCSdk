# Skill Client SDK 项目实施计划（6周 MVP）

## 1. 文档信息
- 文档版本：v1.0（评审稿）
- 事实来源：`documents/skillSdk/SkillClientSdkInterface.md`
- 约束基线：已评审结论（2026-03-07）
- 实施边界：SDK 主导 + Skill Server / IM / 小程序协同项
- 计划周期：6 周

## 2. 实施摘要
本计划目标是在 6 周内完成 Skill Client SDK 文档定稿、主链路实现方案落地、联调验证与发布准备。  
本期不新增业务接口，重点完成契约统一、时序稳定性、错误语义统一和观测能力补齐。

关键定稿约束：
1. `closeSkill()` 无入参，关闭 SDK 管理的全部会话连接并清空 SessionStore 内存上下文。
2. `stopSkill` 语义为“停止本轮生成”，现网仍 `DELETE /sessions/{id}`，MVP 暂不改服务端接口。
3. `SkillSession.id` 与 `userId` 在 SDK 层统一 `string`。
4. `onSessionStatusChange` 不在无 WS 连接时触发自动建连。
5. `StreamMessage.sessionId` 为必填字段。

## 3. 必须落地的接口/类型/指标变更
## 3.1 类型与契约
- `StreamMessage` 新增必填字段：`sessionId: string`
- `SkillSession.id` 类型统一为 `string`
- `userId` 类型统一为 `string`
- 新增统一错误对象：`SkillSdkError`

## 3.2 配置与初始化
- SDK 初始化参数新增：`ConnectionPolicy`
- 默认参数：
  - `maxRetryCount: 5`
  - `backoffInitialMs: 1000`
  - `backoffMaxMs: 5000`
  - `heartbeatIntervalMs: 15000`
  - `disconnectThresholdMs: 30000`
- 支持环境覆写：`dev/test/prod`

## 3.3 观测指标
- 新增指标：`dispatchLatencyMs`（WS 收包到监听器回调完成耗时）
- 继续跟踪：接口成功率、首包时延、回调丢失率、重连次数、权限确认闭环时长

## 7. 依赖与风险管理
## 7.1 外部依赖
- Skill Server：停止语义兼容、分页历史一致性、错误码稳定。
- IM 平台：消息发送可用性与失败码清晰。
- 小程序宿主：窗口控制能力与状态通知可用。

## 7.2 关键风险与应对
- 风险：`stopSkill` 的 DELETE 语义导致会话终止。
  - 应对：SDK 统一输出 `SESSION_TERMINATED_AFTER_STOP`，前端控制体验并提供重建入口。
- 风险：多监听器异常导致事件阻塞。
  - 应对：逐监听器隔离 + 熔断计数，禁止单点回调拖垮全局。
- 风险：弱网重连后消息顺序不一致。
  - 应对：`sessionId + seq` 连续性校验，缺口触发补拉。
