## 1. 分享背景

这次我想分享的，不是“AI 帮我写了多少代码”，而是“我是如何借助 AI，把一个 Skill SDK 项目从需求描述一步步推进到可设计、可评审、可实施状态的”。

这个项目的目标，是围绕 `IM 客户端 + 内嵌小程序 + OpenCode Skill 服务端` 三方协作，设计并推动一套统一的前端 SDK。SDK 主要负责两类能力：

1. Skill 执行过程中的 HTTP / WebSocket 通信。
2. 小程序状态与 IM 客户端之间的回调和协同。

这个项目的事实源主要有两个：

1. 我对项目的需求描述。
2. Skill SDK 和服务端接口文档。

我并没有一开始就让 AI 直接写代码，而是先让 AI 帮我做需求澄清、识别文档冲突、补齐架构设计、输出 API 设计，再进一步形成实施计划和过程文档。也就是说，这次 AI 参与的重点，不只是“编码”，而是参与了项目从想法到研发准备的整个构建过程。

所以这次分享真正想讲的是：

**我是如何把 AI 当成项目构建过程中的协作助手，而不是单纯的代码生成工具。**

## 2. 我是如何使用 AI 的

### 2.1 先让 AI 帮我把问题问清楚，而不是直接写方案

项目一开始，我并没有让 AI 直接输出架构图或者代码，而是先让它扮演“AI 产品经理”，持续追问需求，直到边界足够清晰。

提示语：
你现在是我的产品经理+需求分析师，不是程序员。我只会说模糊想法，你要不断追问，直到需求可落地，无歧义。
规则：
1、不写任何代码；
2、不提供实现方案；
3、只提问，澄清，梳理，确认需求；
4、需求完全明确后，输出一份《产品需求文档》；
明白就回复“已准备好，请说需求”

在这个过程中，AI 帮我不断拆解并确认这些问题：

1. SDK 是给谁用的，是 IM 用，还是小程序用，还是两者共用。
2. Skill 是从哪里触发的，执行入口在哪里。
3. IM 和小程序之间是什么关系，谁负责展示，谁负责处理状态。
4. 会话是怎么标识的，谁生成 `welinkSessionId`。
5. 服务端交互是 HTTP、WebSocket，还是两者并用。
6. 状态回调有哪些，哪些状态需要上报给 IM。
7. 谁来处理并发限制，谁来保证同一聊天窗只执行一个 Skill。
8. SDK 负责哪些事，不负责哪些事。

这一阶段我最大的体会是，AI 最有价值的地方之一，不是立即给答案，而是持续追问，帮助我把模糊的想法变成可落地的问题定义。

### 2.2 让 AI 先读现有文档，再找冲突，再补设计

在需求收敛之后，我没有让 AI 凭空设计，而是要求它先读取已有的 Skill SDK 和服务端接口文档，再结合我的需求描述做比对。

这一步 AI 帮我识别出了很多关键冲突，例如：

1. 服务端文档里写“认证：无”，但真实需求里已经明确需要 token。
2. `executeSkill` 缺少显式 `skillCode` 入参，但业务入口其实就是 `/skillname`。
3. `sessionId` 在文档里出现了 `string/number/Long` 混用，类型不统一。
4. `stopSkill` 的语义前后不一致，描述和实现方式冲突。
5. 小程序关闭后的 SDK 行为描述不一致，有的地方写要关闭会话，有的地方写只回调。

这一步很重要，因为很多项目问题不是“不会设计”，而是“基础文档本身就不一致”。

AI 在这里真正做的，是帮我做“文档审计”和“逻辑一致性检查”。

也正是因为先做了这一步，后面的架构设计和 API 设计才没有建立在错误前提上。

### 2.3 用 AI 把讨论沉淀成正式文档，而不是停留在聊天记录里

这次项目推进过程中，我没有满足于“AI 给了我一个回答”，而是要求它每推进一步，就输出正式文档，并且每份文档都要经过评审。

在这个项目里，AI 最终帮我沉淀了四类关键文档：

1. 需求澄清文档
2. 架构设计文档
3. API 详细设计文档
4. 项目实施计划

后面又进一步整理了项目过程文档。

这种方式的好处很明显：

1. 讨论结果不会散落在聊天记录里，而是会沉淀成项目资产。
2. 每一层文档都有明确职责，需求、架构、接口、实施各自独立。
3. 后续评审有明确对象，不需要再回溯大量上下文。
4. 项目一旦进入研发阶段，团队拿到的是“基线文档”，而不是一堆零散结论。

换句话说，AI 在这里不仅仅是在“回答问题”，而是在帮我把项目的知识结构化。

### 2.4 我不是直接采纳 AI，而是把 AI 放进评审循环里

这次实践里，AI 的输出从来不是“一次生成，直接采用”，而是进入一个持续评审和修订的过程。

最典型的一次是在 API 设计阶段。

AI 先给出了第一版 API 文档，但我在评审时指出了关键问题：

1. `stopSkill` 关闭的是会话，不是 WebSocket 通道。
2. WS 通道应该是 SDK 的全局单实例，并且可复用。
3. `sendMessage` 的返回值应该继续保持 `Promise<boolean>`。

AI 随后根据这轮评审完成修订，把原本不稳定的定义改成了明确结论，最后形成评审通过版文档。

这个过程让我更确定一点：

**AI 很适合先产出高质量草案，但不能跳过评审。**

真正可靠的使用方式，不是让 AI 替我做决定，而是让 AI 先把方案铺出来，再由我做关键确认。

### 2.5 AI 不只是帮我设计，还帮我规划“怎么推进项目”

在这次项目中，AI 不只是参与需求和设计，还进一步参与了项目实施计划的制定。

实施计划并不是简单写一个时间表，而是拆解成了完整的阶段推进方案，包括：

1. 方案冻结
2. 核心底座开发
3. API 全量实现
4. 三方联调闭环
5. 稳定性与验收
6. 灰度发布与全量

在更早的 Skill SDK 文档构建阶段，AI 还帮助把测试门禁做得非常具体，例如：

1. 每个接口要有足够的单元测试覆盖。
2. 要有联动链路测试矩阵。
3. 要有接近真实业务流程的模拟场景测试。
4. 发布前要有明确的质量门禁和回滚条件。

这给我带来的帮助很大，因为很多时候项目难推进，不是因为没人会写代码，而是因为没有明确的执行路径。

AI 在这一点上，实际上帮助我完成了“从设计到落地路径”的连接。

## 3. 这次实践中，我总结出的几条有效方法

### 3.1 把 AI 当成“协作角色集合”，而不是单一写手

这次项目里，AI 实际上承担了多个角色：

1. 产品经理，帮助我做需求澄清。
2. 架构师，帮助我做模块划分和关键时序设计。
3. 技术文档助手，帮助我形成正式设计文档。
4. 评审修订助手，根据评审意见快速收敛文档。
5. 项目助理，帮助我输出实施计划和过程总结。

如果只把 AI 当成代码生成器，会浪费掉它在项目构建早期阶段的大部分价值。

### 3.2 先定义问题，再让 AI 帮你解决问题

这次最明显的感受是：

1. 如果需求没收敛，AI 输出再多内容也容易跑偏。
2. 如果先把边界、角色、状态、约束定义清楚，AI 的产出质量会明显提升。
3. 问题定义越清楚，后面的架构、API、实施计划越稳定。

所以更有效的顺序不是“直接让 AI 出方案”，而是：

1. 先澄清需求
2. 再识别文档冲突
3. 再建立设计基线
4. 最后进入实施规划和研发

### 3.3 用“文档驱动”替代“纯对话驱动”

如果只靠对话推进，一个明显问题是上下文容易丢，团队也很难复用这些结论。

这次更有效的做法是：每完成一轮讨论，就让 AI 把结论沉淀为正式文档，并且让文档具备这些属性：

1. 有版本
2. 有状态
3. 有评审结论
4. 有明确的已确认事项和待确认事项

这样一来，AI 的输出就不只是临时建议，而是变成了真正可复用的项目资产。

### 3.4 AI 最适合做“加速器”，不适合替代“拍板者”

这次项目里，真正需要我拍板的，都是那些影响后续设计和实现的关键点，比如：

1. `stopSkill` 的真实语义
2. WebSocket 生命周期
3. `welinkSessionId` 的统一类型
4. 小程序关闭时是否自动关闭会话
5. `sendMessage` 的返回约定

AI 可以非常快地帮我生成方案、列出选项、整理差异，但最终的关键决策仍然需要人来做。

所以我现在更倾向于这样看待 AI：

**AI 最适合做高质量的方案草拟者和加速器，但项目里的关键决策，仍然应该由人来完成。**

## 4. 通过 AI 构建 Skill SDK 项目，我最终获得了什么

这次实践给我最大的收获，不是“省了多少写文档时间”，而是让我更加清楚 AI 在一个真实项目中的正确位置。

它最适合帮助我完成三件事：

1. **把模糊需求变成清晰边界**  
   通过持续追问、对齐和冲突识别，把原本零散的需求收敛成清晰的项目基线。

2. **把分散讨论变成系统化设计资产**  
   需求文档、架构文档、API 文档、实施计划、过程文档逐步形成，项目从聊天记录变成了正式工程资产。

3. **把个人思考变成可评审、可协作、可落地的推进方式**  
   AI 帮我补全结构、加速整理、协助修订，但关键判断始终掌握在我手里。

如果用一句话总结这次实践，我会这样说：

**我不是让 AI 替我做项目，而是让 AI 帮我更系统、更高效地把项目做出来。**

## 5. 结语

在这个 Skill SDK 项目里，AI 的价值不只是写代码，而是参与了项目从需求澄清、设计沉淀、评审修订，到实施规划的全过程。

它真正帮我解决的，不是“写不写得出来”，而是：

1. 需求能不能更快被定义清楚
2. 文档能不能更快形成基线
3. 项目能不能更稳地进入研发阶段

所以如果要对外分享这次经验，我最想表达的不是“AI 很强”，而是：

**当你把 AI 放进正确的位置，它就能从一个工具，变成项目推进过程中的高效协作者。**
