# 分身客户端会话与资料同步产品需求文档（PRD）

**版本**: v1.0  
**日期**: 2026-03-16  
**状态**: 待确认  
**项目代号**: Clone Client Sync

---

## 一、产品概述

### 1.1 产品定位

本期仅定义分身客户端需要提供的产品能力，不包含 IM 客户端和分身服务端的需求设计。

分身客户端定位为 IM 客户端与分身服务端之间的同步中间层，负责承接数字分身会话设置、分身资料与服务端推送变化的接收、转发、确认、回传及本地状态存储。

### 1.2 默认前提

- IM 客户端已具备会话列表展示、右键操作、资料编辑入口、界面刷新能力
- 分身服务端已具备资料存储、会话状态存储、消息推送、长连接通信能力
- 本期仅针对 Windows IM 客户端场景
- 本期仅针对数字分身单聊场景

### 1.3 核心价值

1. 让 IM 对数字分身会话的操作能够可靠同步到分身侧
2. 让分身服务端的消息、资料与状态变化能够回刷 IM
3. 让分身客户端具备本地会话状态存储能力，支持缓存、离线恢复与状态补齐
4. 保证同一数字分身在 IM 各可见入口上的展示一致

## 二、需求范围

### 2.1 本期范围

- 会话设置同步：置顶、免打扰、标为已读、删除会话
- 分身资料同步：名称、头像、简介
- 服务端推送消费与回传：新消息、资料变化、会话状态变化
- 本地会话状态存储：会话设置状态与分身资料
- 离线后状态补齐

### 2.2 非本期范围

- IM 客户端界面交互设计
- 分身服务端内部处理逻辑设计
- 群聊场景
- 多分身共享一个会话
- 一分身多会话
- 完整消息列表本地持久化
- 分身实体删除、解绑、注销
- 完整资料卡扩展字段

## 三、业务模型与规则

### 3.1 会话模型

- 一个数字分身在 IM 中对应一个唯一单聊会话
- 分身客户端按“一分身一会话”关系维护同步与本地状态

### 3.2 主数据源规则

- 会话设置与分身资料的最终确认结果以分身侧为准
- IM 可先本地立即生效
- 分身客户端必须将服务端最终确认结果回传 IM
- 一旦本地记录与服务端确认结果不一致，以服务端结果覆盖本地

### 3.3 并发规则

- 同一账号多端并发修改同一分身资料或会话设置时，以最后被分身侧确认的结果为准
- 分身客户端收到最终确认结果后，需要同步覆盖本地缓存并回传 IM

### 3.4 删除规则

- 删除会话仅表示移除 IM 中该数字分身的会话入口
- 删除会话不等于删除分身实体，不等于永久清空历史
- 分身客户端本地需保留软删除记录
- 已删除会话仅在以下场景恢复：
  - 收到新消息
  - 用户重新主动发起对话
- 单纯资料变化或静态状态变化，不触发已删除会话恢复
- 已删除会话恢复后，应保留删除前已有的置顶、免打扰等设置

### 3.5 已读规则

- “标为已读”表示 IM 与分身侧该会话未读状态同步归零
- 后续若收到新消息，可重新产生未读状态

### 3.6 资料规则

- 本期纳入同步的资料字段为：名称、头像、简介
- 当前会话用户可发起上述资料修改
- 若资料变化来自分身侧或服务端，而非 IM 发起，分身客户端仍需同步给 IM
- 简介字段本期仅要求用于聊天头部和资料查看区域，不要求进入会话列表摘要

## 四、功能需求

### FR-1 接收 IM 会话操作并同步服务端

分身客户端需要接收 IM 发起的以下会话操作：

- 置顶
- 免打扰
- 标为已读
- 删除会话

功能要求：

- 能识别操作对应的唯一数字分身会话
- 能将操作同步到分身服务端
- 能接收服务端确认结果并回传 IM
- 若同步失败，能返回失败结果，供 IM 回滚并提示用户

### FR-2 接收 IM 资料编辑并同步服务端

分身客户端需要接收 IM 发起的以下资料修改：

- 名称
- 头像
- 简介

功能要求：

- 当前会话用户可发起修改
- 分身客户端负责将修改同步到服务端
- 服务端确认后回传最新资料给 IM
- 若资料变化来自分身侧或服务端，也必须同步给 IM

### FR-3 接收服务端推送并同步 IM

分身客户端与分身服务端保持长连接。

当收到以下变化时，分身客户端需同步给 IM：

- 新消息
- 分身资料变化
- 会话设置变化
- 会话状态变化

同步目标：

- 支撑 IM 刷新左侧会话列表整行信息
- 支撑 IM 刷新当前聊天头部
- 支撑 IM 刷新资料查看区域

### FR-4 本地会话状态存储

分身客户端需对对应数字分身会话状态做本地持久化。

本地存储范围：

- 会话设置状态：置顶、免打扰、已读/未读、删除标记
- 分身资料：名称、头像、简介

本地不要求存储：

- 完整消息列表
- 完整会话快照

业务要求：

- 本地存储用途为缓存与离线恢复，不是最终主数据源
- 首次建立本地记录时，优先以服务端已确认结果初始化
- 删除后保留软删除记录
- 切换账号时按账号隔离存储，互不影响
- 收到服务端确认或补齐结果后，立即覆盖本地缓存

### FR-5 同步失败结果回传

- IM 已本地立即生效时，分身客户端必须提供明确失败反馈
- IM 可据此回滚到上一个已确认状态
- 失败反馈需可区分“未确认成功”与“已确认成功”

### FR-6 离线后状态补齐

若离线期间错过服务端推送，分身客户端恢复后需具备补齐最新确认状态并同步 IM 的能力。

补齐范围：

- 会话设置最终状态
- 分身资料最终状态
- 新消息带来的会话入口恢复或未读变化

## 五、关键场景

### 5.1 IM 右键操作会话

1. IM 发起置顶、免打扰、标为已读或删除操作给分身客户端
2. 分身客户端同步到服务端
3. 服务端确认后，分身客户端更新本地状态并回传 IM
4. 若失败，分身客户端返回失败结果，供 IM 回滚并提示

### 5.2 IM 编辑名称、头像、简介

1. IM 发起资料修改给分身客户端
2. 分身客户端同步到服务端
3. 服务端确认后，分身客户端更新本地资料并回传 IM

### 5.3 服务端主动推送变化

1. 服务端推送新消息、资料变化或会话状态变化
2. 分身客户端更新本地状态
3. 分身客户端将最新结果同步给 IM

### 5.4 已删除会话恢复

1. 分身客户端本地保留软删除记录
2. 当收到新消息或用户重新主动发起对话时，通知 IM 恢复会话入口
3. 恢复后继承原有置顶、免打扰设置

### 5.5 离线恢复

1. 分身客户端恢复与服务端连接
2. 获取离线期间已确认的最新状态
3. 用服务端结果覆盖本地缓存
4. 将补齐后的最终状态同步给 IM

---

## 六、验收标准

- IM 发起的置顶、免打扰、标为已读、删除操作，分身客户端都能同步到服务端并回传确认结果
- IM 发起的名称、头像、简介修改，分身客户端都能同步到服务端并回传最新资料
- 服务端主动变化时，分身客户端能同步 IM 刷新对应数字分身会话
- 分身客户端能对每个数字分身会话持久化本地状态，范围至少覆盖会话设置和资料
- 本地存储用于缓存和离线恢复，且在服务端确认后会被正确覆盖
- 已删除会话不会因单纯资料变化恢复，但会因新消息或用户重新发起对话而恢复
- 已删除会话恢复后，原置顶、免打扰状态仍保留
- 同步失败时，分身客户端能返回失败结果，支持 IM 回滚并提示
- 切换账号后，本地状态按账号隔离，不相互污染
- 离线恢复后，分身客户端能补齐并回传最新确认状态

---

## 七、前提与约束

- 本期只涉及分身客户端能力，IM 与服务端均视为已具备既有能力
- 会话模型采用“一分身一会话”
- 会话设置与资料最终以分身侧确认结果为准
- IM 允许先本地立即生效；若失败，由分身客户端返回失败结果供 IM 回滚
- 本地存储采用“软删除保留、服务端覆盖本地、按账号隔离”的原则
