**版本**: v1.0  
**日期**: 2026-03-16  
**状态**: 待确认  
**项目代号**: Clone Client Sync

---

## 一、架构概览

### 1.1 系统定位

本期架构聚焦分身客户端本体，默认 IM 客户端和分身服务端均为既有系统，不展开其内部实现。

分身客户端作为独立客户端进程，位于 IM 客户端与分身服务端之间，负责以下职责：

- 接收 IM 发起的会话设置与资料修改命令
- 同步命令到分身服务端并接收确认结果
- 接收分身服务端主动推送的消息、资料与会话状态变化
- 对每个数字分身会话维护内存态与本地持久化态
- 向 IM 输出领域事件与最新确认快照

### 1.2 设计原则

1. **服务端权威，客户端缓存**：本地状态用于缓存和恢复，不作为最终权威来源。
2. **单连接承载多会话**：一个客户端只维持一条面向服务端的 WebSocket 长连接，承载当前账号下所有数字分身会话事件。
3. **启动先快照，运行期靠增量**：IM 启动或重连后先拉全量快照，再订阅增量事件。
4. **事件与快照并存**：分身客户端既输出变化事件，也输出最新确认快照，降低 IM 侧状态归并复杂度。
5. **软删除保留**：删除会话只移除入口，不清理本地核心状态，便于后续恢复。
6. **账号隔离存储**：不同账号的本地状态完全隔离，不共享存储。
7. **失败即时返回**：本期不同步失败时不做后台重试，直接返回 IM，供其回滚和提示。

### 1.3 外部边界

| 外部系统 | 角色 | 分身客户端依赖内容 |
|------|------|------|
| IM 客户端 | 上游调用方 | 发起命令、拉取快照、订阅事件、刷新 UI |
| 分身服务端 | 下游权威源 | 接收命令、返回确认、提供快照、通过 WebSocket 推送增量事件 |

## 二、整体架构

### 2.1 集成架构图

```text
┌────────────────────┐      Local IPC / Local API      ┌─────────────────────────┐
│ IM Client          │ <-----------------------------> │ Clone Client            │
│ - 会话列表刷新      │                                 │ - Local API Facade      │
│ - 聊天头部刷新      │                                 │ - Sync Orchestrator     │
│ - 资料页刷新        │                                 │ - Server Conn Manager   │
│ - 发起设置/资料修改 │                                 │ - Server Event Router   │
└────────────────────┘                                 │ - Conversation State    │
                                                       │ - Snapshot Builder      │
                                                       │ - Event Publisher       │
                                                       └──────────┬──────────────┘
                                                                  │
                                                                  │ REST / WebSocket
                                                                  ▼
                                                       ┌─────────────────────────┐
                                                       │ Clone Server            │
                                                       │ - 状态权威源            │
                                                       │ - 命令确认              │
                                                       │ - 全量快照              │
                                                       │ - 增量事件推送          │
                                                       └─────────────────────────┘
```

### 2.2 模块划分

| 模块 | 职责 |
|------|------|
| `Local API Facade` | 对 IM 暴露命令、查询、订阅三类本地接口，屏蔽内部编排细节 |
| `Sync Orchestrator` | 统一编排 IM 指令下发、服务端确认、失败回传、状态覆盖与事件发布 |
| `Server Connection Manager` | 管理账号级单 WebSocket 连接、连接状态、重连后的补齐触发 |
| `Server Event Router` | 解析服务端推送并分流为消息变化、资料变化、会话设置变化、状态变化 |
| `Conversation State Store` | 维护内存态 + 本地持久化态，按账号隔离保存会话设置与资料 |
| `Snapshot Builder` | 基于当前状态生成给 IM 的最新确认快照 |
| `Event Publisher` | 向 IM 发布领域事件和对应的最新快照 |

### 2.3 不单列消息持久化模块

本期不要求本地持久化完整消息列表，因而架构上不单列消息持久化模块。  
消息相关能力仅体现在：

- 接收服务端“新消息到达”事件
- 基于该事件驱动会话恢复、未读变化和列表刷新

## 三、核心模块设计

### 3.1 Local API Facade

对 IM 提供统一本地接口，分为三组：

- 命令接口：处理会话设置修改与资料修改
- 查询接口：返回当前账号下全部会话快照或单会话快照
- 订阅接口：向 IM 持续推送会话变化事件

职责约束：

- 不承担业务状态归并
- 不直接访问服务端
- 所有业务处理统一下沉到 `Sync Orchestrator`

### 3.2 Sync Orchestrator

该模块是架构主编排层，统一处理两类输入：

- IM 发起的命令
- 服务端主动推送的增量事件

其主要职责为：

- 将 IM 命令转换为服务端命令调用
- 接收服务端确认结果
- 依据服务端确认顺序更新本地状态
- 构造最新确认快照
- 向 IM 发布成功事件或失败事件

关键约束：

- 失败后不做后台重试
- 成功与失败都必须有明确回传结果
- 只以服务端确认或推送结果更新最终状态

### 3.3 Server Connection Manager

负责当前账号与分身服务端之间的单 WebSocket 长连接管理。

职责包括：

- 建立、关闭、重建单连接
- 感知连接状态变化
- 在重连成功后触发一次状态补齐
- 将原始推送消息转交 `Server Event Router`

关键约束：

- 一个客户端只维护一条连接
- 不按数字分身、会话维度重复建连
- 连接恢复后先补齐再发布恢复后的稳定状态

### 3.4 Server Event Router

负责解析服务端推送，并按业务类型分发到编排层。  
至少支持以下事件类别：

- 新消息到达
- 分身资料变化
- 会话设置变化
- 会话状态变化

关键职责：

- 识别事件归属的 `accountId / cloneId / conversationId`
- 生成可供 `Sync Orchestrator` 消费的内部领域事件
- 保持事件顺序与服务端确认顺序一致

### 3.5 Conversation State Store

负责状态存储，是架构中的本地状态中心。  
状态分为两层：

- 内存态：运行期快速读写、事件分发、快照组装
- 持久化态：启动恢复、离线补齐、软删除保留

核心职责：

- 按账号隔离维护状态
- 按会话标识定位记录
- 保存会话设置状态与分身资料
- 保留软删除记录
- 在服务端确认后覆盖本地缓存

### 3.6 Snapshot Builder

负责从当前状态中生成 IM 可直接消费的最新确认快照。  
快照目标不是承载全部领域细节，而是保证 IM 能直接刷新：

- 左侧会话列表
- 当前聊天头部
- 资料查看区域

### 3.7 Event Publisher

负责把领域事件与对应快照发布给 IM。  
每次发布都遵循：

- 先确定最新状态
- 再构造最新快照
- 再推送事件与快照

这样可保证 IM 不需要自行做复杂状态归并。

## 四、本地状态模型

### 4.1 状态对象最小字段

| 字段 | 含义 |
|------|------|
| `accountId` | 当前账号标识 |
| `cloneId` | 数字分身标识 |
| `conversationId` | 数字分身在 IM 中对应的唯一单聊会话标识 |
| `pinned` | 置顶状态 |
| `muted` | 免打扰状态 |
| `readState` | 已读/未读状态 |
| `deleted` | 是否处于软删除状态 |
| `profile.name` | 分身名称 |
| `profile.avatar` | 分身头像 |
| `profile.bio` | 分身简介 |
| `lastConfirmedSeq` 或 `version` | 最近一次被服务端确认的顺序号/版本号 |
| `lastUpdatedAt` | 最近一次状态更新时间 |

### 4.2 存储层次

| 层次 | 作用 |
|------|------|
| 内存态 | 运行期快速读取、事件发布、快照构建 |
| 持久化态 | 启动恢复、连接恢复、离线补齐、软删除保留 |

### 4.3 存储规则

- 首次建立本地记录时，优先以服务端已确认结果初始化
- 收到服务端确认或补齐结果后，立即覆盖内存态与持久化态
- 删除会话后不物理清空核心状态，保留软删除记录
- 切换账号时按账号维度隔离存储
- 本地存储不是主数据源，只承担缓存和恢复职责

## 五、本地接口定义

### 5.1 查询接口

#### `getAllConversationSnapshots(accountId)`

语义：

- 返回当前账号下全部数字分身会话的最新确认快照
- 用于 IM 启动或重连后的全量初始化

#### `getConversationSnapshot(accountId, conversationId)`

语义：

- 返回指定会话的最新确认快照
- 用于 IM 按需读取单会话状态

### 5.2 订阅接口

#### `subscribeConversationEvents(accountId, callback)`

语义：

- 订阅当前账号下数字分身会话的变化事件
- 事件流中每条事件都带对应会话的最新确认快照

### 5.3 命令接口

#### `updateConversationSetting(accountId, conversationId, action, value)`

语义：

- 处理置顶、免打扰、标已读、删除会话等命令
- 命令成功后发布会话设置变更事件
- 命令失败后发布失败事件并将失败结果返回 IM

#### `updateCloneProfile(accountId, cloneId, profilePatch)`

语义：

- 处理名称、头像、简介修改
- 服务端确认成功后发布资料变更事件
- 服务端确认失败后发布失败事件并将失败结果返回 IM

## 六、领域事件与快照模型

### 6.1 事件类型

分身客户端至少向 IM 输出以下事件：

- `conversation_setting_changed`
- `profile_changed`
- `conversation_restored`
- `server_message_arrived`
- `sync_failed`

### 6.2 事件与快照关系

- 事件表示“发生了什么变化”
- 快照表示“该会话当前最新确认状态是什么”
- 每个事件都附带对应会话的最新确认快照

### 6.3 快照作用

IM 基于快照直接刷新：

- 左侧会话列表
- 当前聊天头部
- 资料查看区域

因此 IM 不需要自行维护复杂的状态归并逻辑。

---

## 七、状态覆盖规则

### 7.1 覆盖优先级

1. 服务端确认顺序优先
2. 服务端结果覆盖本地缓存
3. 本地缓存仅用于恢复，不反向覆盖服务端结果

### 7.2 删除与恢复规则

- 已删除会话只在“新消息到达”或“用户重新主动发起对话”时恢复
- 单纯资料变化不触发恢复
- 恢复时保留原置顶、免打扰等设置

### 7.3 失败规则

- IM 发起操作失败时，分身客户端返回失败结果
- 本期不引入后台重试队列
- IM 依据失败结果自行回滚界面状态

## 八、关键时序与数据流

### 8.1 IM 启动或重连

1. IM 调用 `getAllConversationSnapshots(accountId)`
2. 分身客户端从本地状态中心读取当前账号全部会话快照
3. 若处于启动恢复或连接恢复阶段，先完成一次服务端状态补齐
4. 分身客户端返回全量快照给 IM
5. IM 调用 `subscribeConversationEvents(accountId, callback)` 订阅后续增量事件

### 8.2 IM 发起会话设置修改

1. IM 调用 `updateConversationSetting(...)`
2. `Local API Facade` 将请求交给 `Sync Orchestrator`
3. `Sync Orchestrator` 调用服务端命令接口
4. 服务端返回确认结果
5. 若成功：
   - 更新本地状态
   - 生成最新快照
   - 发布 `conversation_setting_changed`
6. 若失败：
   - 返回失败结果
   - 发布 `sync_failed`

### 8.3 IM 发起资料修改

1. IM 调用 `updateCloneProfile(...)`
2. 分身客户端调用服务端命令接口
3. 服务端返回确认结果
4. 若成功：
   - 更新本地资料
   - 生成最新快照
   - 发布 `profile_changed`
5. 若失败：
   - 返回失败结果
   - 发布 `sync_failed`

### 8.4 服务端主动推送

1. `Server Connection Manager` 通过单连接接收推送
2. `Server Event Router` 解析原始事件并路由到 `Sync Orchestrator`
3. `Sync Orchestrator` 按服务端确认顺序更新本地状态
4. `Snapshot Builder` 生成最新快照
5. `Event Publisher` 向 IM 发布对应事件与快照

### 8.5 已删除会话恢复

1. 分身客户端收到新消息事件或用户重新发起对话相关确认
2. `Sync Orchestrator` 检查会话是否处于软删除状态
3. 若满足恢复条件：
   - 清除删除标记
   - 保留原置顶、免打扰设置
   - 生成最新快照
   - 发布 `conversation_restored`

### 8.6 离线恢复

1. 单连接恢复
2. `Server Connection Manager` 触发补齐流程
3. 分身客户端向服务端拉取当前账号下最新确认状态
4. 服务端结果覆盖本地缓存
5. 分身客户端按最新状态生成快照并向 IM 发布恢复后的增量事件

### 8.7 明确不支持的触发

- 仅资料变化不恢复已删除会话
- 命令失败后不自动重试

## 九、关键架构决策

### 9.1 独立进程而非 IM 内模块

采用独立分身客户端进程，便于明确本地状态边界、服务端连接边界与 IM 本地接口边界。

### 9.2 单 WebSocket 连接模型

采用账号级单连接，避免按分身或会话重复建连，降低连接数与状态同步复杂度。

### 9.3 启动先全量快照，再订阅增量

这样可确保 IM 在启动时立即拿到当前稳定状态，不必等待增量事件逐步收敛。

### 9.4 输出“事件 + 最新快照”

这样可将状态归并复杂度收敛在分身客户端，降低 IM 侧实现成本与出错概率。

### 9.5 本地存储只做缓存与恢复

本地状态不是主数据源，所有最终状态均以服务端结果为准，避免多源冲突。

## 十、约束与风险

### 10.1 约束

- 本期不设计后台重试
- 本期不设计消息本地持久化
- 本期不覆盖群聊、多会话模型
- 本期不锁定具体存储引擎与 IPC 技术，只锁定能力边界与数据流

### 10.2 风险

- 单连接中断期间会形成状态补齐窗口
- 服务端确认顺序可能与 IM 本地直觉不一致，导致用户看到“先变后回滚”的体验
- 软删除记录长期保留会带来后续数据清理策略问题
- 事件风暴场景下，IM 侧刷新频率控制需要与分身客户端接口节奏共同约束

---

## 十一、架构验证清单

- IM 启动后能一次性拿到当前账号全部数字分身会话快照
- 置顶、免打扰、标已读、删除在成功和失败链路下都有明确处理
- 名称、头像、简介既支持 IM 发起，也支持服务端外部变更回刷
- 已删除会话只在允许的触发条件下恢复
- 单连接能承载同一账号下所有数字分身事件
- 切换账号后本地状态不串号
- 离线恢复后本地缓存会被服务端最新确认状态覆盖
- IM 只消费事件与快照即可刷新，无需自行推导最终状态
